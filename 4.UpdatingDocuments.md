## 🔧 The `$set` Operator

Before we start updating, let's understand the most important update operator: **`$set`**.

`$set` does two things:
- **Creates** a field if it doesn't exist
- **Updates** the value if it does exist

Without `$set`, MongoDB would try to **replace** the entire document — which is almost never what you want. Always use `$set` for targeted updates!

```javascript
// ✅ Correct — updates just the age field
{ $set: { age: 24 } }

// ❌ Wrong — would REPLACE the entire document with just { age: 24 }
// (losing name, phone, role, etc.)
{ age: 24 }
```

## 📄 Updating a Single Document — `updateOne()`

Let's add social media info to Gebin's document. The `updateOne()` method takes two arguments:

1. **Filter** — which document to update
2. **Update** — what changes to make

```javascript
db.mydata.updateOne(
  { name: "Gebin" },
  { $set: { socials: { hasGithub: true, hasLinkedin: true } } }
)
```

**Output:**
```
{
  acknowledged: true,
  insertedId: null,
  matchedCount: 1,
  modifiedCount: 1,
  upsertedCount: 0
}
```

> **What happened here?**
> - We told MongoDB: _"Find the document where `name` is `Gebin`, and add a `socials` field to it"_
> - The `socials` field is a **nested document** — an object inside our document
> - `matchedCount: 1` → MongoDB found 1 document matching our filter
> - `modifiedCount: 1` → It successfully modified that document

## ✅ Verifying Your Update

Always a good habit to check that the update worked:

```javascript
db.mydata.findOne({ name: "Gebin" })
```

**Output:**
```
{
  _id: ObjectId('699d49a83321373e437c2907'),
  name: 'Gebin',
  age: 23,
  phone: '9876543210',
  role: 'FullStackDev',
  socials: { hasGithub: true, hasLinkedin: true }
}
```

The `socials` field has been added as a **nested document** (an object inside our document). MongoDB handles this natively — no special configuration needed.

## 📚 Updating Multiple Documents — `updateMany()`

What if we want to add the same `socials` field to **every** document in the collection? That's where `updateMany()` comes in:

```javascript
db.mydata.updateMany(
  {},
  { $set: { socials: { hasGithub: true, hasLinkedin: true } } }
)
```

> **📌 Important:** The empty filter `{}` matches **all documents** in the collection. This is powerful but dangerous — always double-check your filter before running `updateMany()`!

**Output:**
```
{
  acknowledged: true,
  insertedId: null,
  matchedCount: 5,
  modifiedCount: 4,
  upsertedCount: 0
}
```

> **Why `modifiedCount: 4` and not 5?** Because Gebin's document already had the `socials` field from our earlier `updateOne()` — MongoDB is smart enough to skip documents that already have the exact same value.

## 📝 Adding Arrays to Documents

MongoDB handles **arrays** natively, which is one of the things that makes it so flexible compared to traditional SQL databases. Let's give everyone an `education` field:

```javascript
db.mydata.updateMany(
  {},
  { $set: { education: ["BCA", "MCA"] } }
)
```

Now every document in the collection has an `education` array:

```javascript
db.mydata.findOne({ name: "Gebin" })
```

**Output:**
```
{
  _id: ObjectId('699d49a83321373e437c2907'),
  name: 'Gebin',
  age: 23,
  phone: '9876543210',
  role: 'FullStackDev',
  socials: { hasGithub: true, hasLinkedin: true },
  education: [ 'BCA', 'MCA' ]
}
```

> **💡 Fun fact:** In SQL, storing an array of values would require a **separate table** and a **JOIN**. In MongoDB, arrays are first-class citizens — just another field type!

---

## 🔍 Understanding the Output

Here's what each field in the update response means:

| Field | Meaning |
|:------|:--------|
| `acknowledged: true` | Server received and processed the operation |
| `insertedId: null` | No new document was created (it's `null` for updates) |
| `matchedCount` | Number of documents that matched the filter |
| `modifiedCount` | Number of documents that were actually changed |
| `upsertedCount` | Number of documents inserted via upsert (0 unless you use `upsert: true`) |

> **🧠 `matchedCount` vs `modifiedCount`:** If a document matches the filter but already has the exact value you're trying to set, it will be _matched_ but not _modified_. This is why these two numbers can differ.

---

## 📋 Quick Reference

| Command | What It Does |
|:--------|:-------------|
| `db.<col>.updateOne({filter}, {$set: {...}})` | Update the first matching document |
| `db.<col>.updateMany({filter}, {$set: {...}})` | Update all matching documents |
| `db.<col>.updateMany({}, {$set: {...}})` | Update ALL documents (empty filter) |
| `$set` | Create or update specific fields |
| `{ key: { nested: value } }` | Store nested documents (objects) |
| `{ key: [value1, value2] }` | Store arrays |
